## 🎮 这项目到底在做什么？

做一个**俯视角战棋游戏**。类似《火焰纹章》那种：
- 鼠标悬停在格子上，格子会亮
- 点击空地，角色沿着格子走过去
- 有墙的地方角色绕过去，不会穿墙

## 第一课：鼠标悬停高亮

### 问题是啥？

画面里铺了一堆瓦片（像地砖），鼠标移到哪块砖上，要在那块砖正中央显示一个高亮光圈。

### 难点：鼠标知道的是"像素"，地砖用的是"第几块"

想象你站在一个广场上：
- **像素坐标**："我在东经 116.4° 北纬 39.9°"（很精确，但没法直接说在哪块砖）
- **网格坐标**："我在第 3 行第 5 列的地砖上"（整数，游戏逻辑用的）

所以需要**翻译官**把两种坐标互相转换。

### 三个脚本的分工（像一家餐厅）

| 脚本 | 角色 | 干啥 |
|------|------|------|
| `nav_layer.gd` | **地砖厂** | 铺地砖的，知道每块砖多大 |
| `grid_manager.gd` | **翻译官** | 专门换算"像素 ↔ 第几块砖" |
| `grid_indicator.gd` | **手电筒** | 一个带光圈图片的节点，跟着鼠标跑 |

### 翻译官的换算公式（背下来）

```
像素 → 第几块砖：
  先问地砖厂"鼠标相对于你在哪" → 再除以每块砖的大小
  
  nav_layer.to_local(鼠标像素)  →  nav_layer.local_to_map(结果)

第几块砖 → 像素：
  先乘砖的大小 → 再问地砖厂"相对于整个世界在哪"
  
  nav_layer.map_to_local(砖号)  →  nav_layer.to_global(结果)
```

**为什么要先 `to_local`？**

地砖厂可能不在广场原点（被挪到角落了）。你站在广场上指一个点，想知道是地砖厂的第几块砖，必须先把坐标转成"相对于地砖厂的角度"。

> 现在地砖厂在原点，所以 `to_local` 看起来啥都没做。但以后换关卡、移动地图，没这步就全错了。这叫**防御性编程**。

### 手电筒怎么跟着走？

每帧（60 次/秒）问翻译官：
1. 鼠标现在在第几块砖？
2. 这块砖的像素中心在哪？
3. 把手电筒移到那个像素位置

**优化**：如果鼠标还在同一块砖里晃，砖号没变，手电筒就不动。省性能。

### 一句话记住第一课

> **像素是精确位置，砖号是游戏逻辑。翻译官负责来回换算，手电筒负责显示。**

## 第二课：角色点击移动 + A* 寻路

### 问题是啥？

点击空地，角色要沿着格子走过去。不能直接飞过去，得沿着格子一步一步走，还要走最短路线。

### A* 是什么？

**手机导航软件。**

你输入"从家到公司"，导航软件自动找最短路线。A* 就是游戏里的导航软件，只不过它的"路"是格子。

### 导航软件怎么初始化？（`nav_layer.gd` 里的 `initialize()`）

```
创建导航软件 AStarGrid2D
├── 告诉它地图范围有多大（`get_used_rect()`）
├── 告诉它每块砖多大（16×16 像素）
├── 告诉它能不能斜着走（不能，战棋只走上下左右）
└── `update()` → 生成内部地图数据
```

### 角色移动的三个步骤

**第一步：点击 → 算路线**

```
玩家点击鼠标
  ↓
角色问翻译官：鼠标在第几块砖？
  ↓
角色说：导航软件，从我现在的砖到鼠标那块砖，最短路线是？
  ↓
导航软件返回一串砖号：[砖1, 砖2, 砖3, 砖4]
  ↓
翻译成像素坐标：[像素1, 像素2, 像素3, 像素4]
  ↓
存进 `path` 数组
```

**第二步：每帧沿着路线走**

`path` 就像地上的一排脚印：

```
path = [👣, 👣, 👣, 👣]
         ↑
       当前目标
```

角色每帧盯着第一个脚印 `path[0]` 走，走到后擦掉它（`remove_at(0)`），跟着下一个走。

**第三步：具体移动**

`move_toward(目标, 速度 × 时间差)`

- 朝目标平滑移动
- 如果剩余距离比这一步还小，直接跳到目标（不会冲过头）
- `速度 × 时间差` 保证走多快和电脑卡不卡没关系

### `_unhandled_input` 是干嘛的？

点击分优先级：
1. UI 先吃（点按钮、点菜单）
2. UI 没处理的，才轮到游戏世界

这样点"结束回合"按钮时，角色不会跟着乱走。

### 一句话记住第二课

> **点击 → 导航算路线 → 存成一串像素点 → 每帧踩着脚印走。**

## 第三课：避障 + 格子档案系统

### 问题是啥？

第二课所有格子都能走，角色会穿墙。现在要让墙变成"走不通"。

### 怎么标记墙？

**方法一：给瓦片贴标签**

`NaviData.png` 有两块瓦片：
- 第一块（0:0）贴了标签 `walkable = true` → 地板
- 第二块（1:0）没贴标签 → 默认 `walkable = false` → 墙

在编辑器里用不同瓦片刷地图，地板和墙就自带属性了。

**方法二：给每块砖建档案卡（GridData）**

```
GridData 档案卡模板
├─ walkable: true/false   ← 能走吗？
└─ occupied_unit: 谁站着  ← 有人吗？
```

游戏启动时，遍历所有画了瓦片的格子，给每个格子发一张档案卡，存进 `grid_data_dict`（档案柜）。

```
档案柜 grid_data_dict：
  钥匙：第几块砖（Vector2i）
  内容：那张砖的档案卡（GridData）
```

### 遇到墙怎么办？

启动时检查：如果瓦片的 `walkable = false`：
1. 档案卡写上 `walkable = false`
2. 告诉导航软件：`set_point_solid(这块砖)` → 导航绕开这块砖

```
地图：
┌──┬──┬──┬──┐
│起│  │墙│墙│
├──┼──┼──┼──┤
│  │  │墙│目│
└──┴──┴──┴──┘

导航算出来的路：下 → 下 → 右 → 右 → 上 → 上（绕开墙）
```

### GridManager 新增的查询功能

| 功能 | 干嘛用的 |
|------|---------|
| `is_valid_grid` | 鼠标点地图外面了？返回 false |
| `is_grid_walkable` | 这格能走吗？ |
| `set_grid_walkable` | 动态改（比如门开了），同时更新导航 |
| `is_grid_occupied` | 这格有人吗？（战棋不能两个人站一起） |
| `set_grid_occupied` | 角色移动时，旧格清空，新格登记 |

### 为什么要 GridData + A* 分开？

- **A*（导航软件）**：只管寻路，不知道这格是墙还是水还是陷阱
- **GridData（档案卡）**：存详细信息，以后能扩展：移动力消耗、陷阱伤害、地形加成……

两者解耦：改档案 → 同步更新导航。其他逻辑只查档案，不直接碰导航。

### 一句话记住第三课

> **每块砖一张档案卡，启动时扫描全图建档。墙在档案上标"不可走"，同时让导航绕开。**

## 第四课：技能系统（初步骨架）

### 问题是啥？

第二节课 Unit 直接处理移动，所有逻辑挤在一起。以后加了攻击、施法、喝药……Unit 会变成一团乱麻。

### 解决方案：把"做什么"和"怎么做"分开

| 角色 | 职责 |
|------|------|
| **Unit** | 收到玩家输入，"我要移动" |
| **ActionsManager（外卖平台）** | 登记所有厨师，按编号找人 |
| **BaseAction（厨师模板）** | 统一接口：接单 → 干活 → 炒完通知 |
| **MoveAction/AttackAction（具体厨师）** | 继承 BaseAction，实现各自的干活逻辑 |

### Callable：把函数当电话号码存起来

正常情况下函数是"直接执行"：
```gdscript
func 做菜():
    print("好了")
做菜()  # 立刻执行
```

Callable 是"先存起来，以后拨打"：
```gdscript
var 电话: Callable = 做菜    # 存进通讯录（不执行）
# 一小时后...
电话.call()                   # 现在拨打，执行！
```

**为什么要存起来？**

厨师（BaseAction）炒菜时，外卖平台（ActionsManager）不想一直盯着。平台给了厨师一个电话，厨师炒完自己拨过去。

### BaseAction：厨师模板

```gdscript
class_name BaseAction

@export var action_id:String        # 厨师编号，如 "move"
@export var action_name:String      # 厨师名字，如 "移动"
var unit:Unit                       # 在哪个店（Unit）工作
var is_active:bool = false          # 正在炒菜吗？
var on_action_finished:Callable     # 炒完打这个电话
```

**start_action（接单）：**
```gdscript
func start_action(目标格子, 电话号码:Callable):
    is_active = true
    self.on_action_finished = 电话号码   # 存进通讯录
```

**finish_action（通知）：**
```gdscript
func finish_action():
    is_active = false
    on_action_finished.call()          # 拨打存好的电话
```

### ActionsManager：外卖平台

**开店时登记厨师：**
```gdscript
func _ready():
    for 厨师 in get_children():        # 遍历所有子节点
        actions.append(厨师)           # 登记到花名册
```

**按编号找厨师：**
```gdscript
func get_action(action_id):
    var results = actions.filter(func(a): return a.action_id == action_id)
    if not results.is_empty():
        return results[0]              # 返回找到的厨师
    return null
```

### 为什么加 self.？

```gdscript
self.on_action_finished = on_action_finished
```

参数名和成员变量名一样，都是 `on_action_finished`。不加 `self` 等于"把参数赋值给参数"，什么都没发生。加了 `self` 才是"把参数存到成员变量"。

### 一句话记住第四课

> **Unit 只管点菜，ActionsManager 管找厨师，厨师炒完自己打电话。新增技能 = 新增一个厨师，不改平台。**

## 第五课：移动技能 + 其他技能模板

### 问题是啥？

第四课搭了空骨架，Unit 还是自己在走路。现在要把移动逻辑真正拆进 MoveAction，让 Unit 只负责"喊开工"。

### Unit 以前 vs 现在

**以前（第二课）：**
```
Unit 自己管一切：
├── 收到点击
├── 算路径
├── 每帧走路
└── 擦脚印
```

**现在（第五课）：**
```
Unit 只管喊开工：
├── 收到点击
├── is_performing_action = true（上锁，防止连点）
├── actions_manager.get_action("move_action").start_action(目标, 回调)
└── 干等

MoveAction 管走路：
├── start_action 里算路径
├── _process 每帧走路
└── 走完了 finish_action() → 拨通 Unit 的电话

Unit 接电话：
└── on_action_finished() → is_performing_action = false（解锁）
```

### 状态锁 `is_performing_action`

战棋是**一个指令执行完才能下另一个**。

```
玩家狂点鼠标：
第1次点击 → 开始移动 → is_performing_action = true
第2次点击 → _unhandled_input 看到 true → 直接 return（屏蔽）
第3次点击 → 同上，被屏蔽
...
移动结束 → on_action_finished() → is_performing_action = false → 才能接受下一次
```

### `super.start_action(...)` 是干嘛的？

父类 BaseAction 已经写了开工的标准流程：
```gdscript
func start_action(target, on_action_finished):
    is_active = true
    self.on_action_finished = on_action_finished
```

MoveAction 不想重写一遍，所以：
```gdscript
func start_action(target, on_action_finished):
    super.start_action(target, on_action_finished)   # 先执行父类的标准流程
    path = GridManager.get_nav_world_path(...)        # 再追加自己的逻辑（算路径）
```

**`super` = "调用父类的同名函数"。**

### `unit.global_position`

MoveAction 不是 Unit 本身，是挂在 ActionsManager 下的独立节点。所以要通过 `unit` 引用（BaseAction 从 `owner` 获取的）来操作角色的位置。

```
MoveAction 节点 ──unit──→ Unit 节点
```

### `_process` 里的 `if not is_active: return`

保险栓。只有 `start_action` 被调用后（`is_active = true`），MoveAction 才会执行移动逻辑。平时进来直接 return，什么都不做。

### 其他技能模板

BowAction、SwordAction、FireballAction 目前只打印一行字就结束。但它们的结构已经搭好了：

```gdscript
extends BaseAction
class_name BowAction

func start_action(target, on_action_finished):
    super.start_action(target, on_action_finished)
    # 以后在这里写：播放动画、生成箭矢、检测命中、扣血
    finish_action()
```

**以后把 `print` 换成实际逻辑就行，Unit 和 ActionsManager 完全不用改。**

### Callable 名片的本质

Unit 传给 MoveAction 的不是"函数的名字"，而是一张**包含回家地址的名片**：

```
┌─────────────────────────┐
│  Callable 名片           │
├─────────────────────────┤
│ 对象: Unit (self)       │  ← 回家地址
│ 函数: on_action_finished │  ← 电话号码
└─────────────────────────┘
```

MoveAction 存好名片，走完后 `on_action_finished.call()` → Godot 按名片上的地址找到 Unit，在其身上执行函数。

**不是跨脚本传播，是函数引用自带对象地址。**

### 一句话记住第五课

> **Unit 上锁→喊开工→MoveAction 走路→走完了拨名片上的电话→Unit 解锁。新增技能改自己的脚本就行，外面不动。**

## 第六课：技能UI面板

### 问题是啥？

第五课技能做好了，但玩家看不到自己有什么技能。现在要把 ActionsManager 里的技能列表，变成屏幕上能看到的按钮。

### 三个新东西的分工

| 东西 | 角色 | 干啥 |
|------|------|------|
| `game_ui.tscn` | **UI总层** | CanvasLayer，所有UI的容器，不受相机影响 |
| `UnitActionUI` | **面板脚本** | 从 ActionsManager 读取技能列表，生成一排按钮 |
| `ActionCardUI` | **单个按钮** | 继承 Button，显示技能名字，以后点了触发技能 |

### PackedScene = 图纸

```gdscript
@export var action_card_ui_scene:PackedScene
```

`PackedScene` 不是真的按钮，是**按钮的图纸/模具**。就像乐高说明书，按着它可以拼出无数个相同的按钮。

```
PackedScene（图纸）
    ↓ instantiate()
ActionCardUI 实例1（真的按钮）
ActionCardUI 实例2（真的按钮）
ActionCardUI 实例3（真的按钮）
```

### 生成按钮的流程

```gdscript
func update_unit_actions_ui():
    # 1. 清空旧按钮（防止重复生成）
    for node in action_container.get_children():
        node.queue_free()
    
    # 2. 遍历所有技能，每个技能造一个按钮
    for action in actions_manager.actions:
        var btn = action_card_ui_scene.instantiate()  # 从图纸造一个
        action_container.add_child(btn)               # 塞进横向排列的盒子
        btn.set_up(action)                            # 设置文字 = 技能名字
```

| 代码 | 类比 |
|------|------|
| `instantiate()` | 按图纸造出一个真的 |
| `add_child()` | 把造好的东西放进展示柜 |
| `set_up(action)` | 贴上标签："这是移动技能" |
| `queue_free()` | 把旧的丢掉 |

### 为什么用 `call_deferred`？

```gdscript
func _ready():
    call_deferred("update_unit_actions_ui")
```

`_ready()` 时节点树可能还没完全搭好，直接执行可能找不到 ActionsManager。`call_deferred` = "等一帧，所有节点都就位了再执行"。

### 场景结构

```
LevelScene
├── GameUI (CanvasLayer)              ← UI总层，永远在最上面
│   └── UnitActionsUI (MarginContainer)
│       └── MarginContainer
│           └── ActionContainer (HBoxContainer) ← 按钮横着排
│               ├── [移动] [弓箭] [剑击] [火球]
```

### 一句话记住第六课

> **ActionsManager 管"有哪些技能"，UnitActionUI 管"把技能变成按钮显示出来"，ActionCardUI 是单个按钮的模板。 PackedScene 是图纸，instantiate() 是造出来。**

## 第七课：技能选择（大重构）

### 问题是啥？

第五课 Unit 自己管输入、管状态锁、管移动，太累了。而且以后多个角色时，每个 Unit 都自己管输入会冲突。

### 解决方案：抽一个全局前台

```
以前（第五课）：
Unit ──→ 自己收输入 → 自己上锁 → 自己调用 MoveAction → 自己解锁

现在（第七课）：
PlayerActionManager（全局前台）──→ 收输入 → 上锁 → 调用当前选中的 Action → 解锁
              ↑
              └── UI 按钮点击 → set_selected_action("弓箭")
```

### PlayerActionManager 三个核心函数

| 函数 | 人设 | 什么时候被调用 |
|------|------|---------------|
| `set_selected_action` | **技能切换台** | 点了 UI 按钮（"我要用弓箭"） |
| `try_perform_selected_action` | **执行器** | 点了鼠标左键（"对目标格子使用当前技能"） |
| `on_action_finished` | **收工通知** | 技能执行完，回调到这里（"干完了，可以下一个"） |

### Unit 彻底解放

```gdscript
# 以前（第五课）：Unit 自己管输入、状态锁、移动
# 现在（第七课）：只剩两件事
extends Node2D
class_name Unit

@onready var actions_manager: ActionsManager = $ActionsManager
var grid_position:Vector2i:
    get:return GridManager.get_grid_position(global_position)
```

删掉了：`_unhandled_input`、`is_performing_action`、`path`、`move_speed`、`target_global_position`、`_process`

### 点了按钮怎么换技能？

```gdscript
# ActionCardUI
func _ready():
    pressed.connect(on_action_seleted)   # 按钮被按下时，执行这个函数

func on_action_seleted():
    PlayerActionManager.set_selected_action(action)  # 通知前台："我选了这个技能"
```

**信号连接**：Button 有内置信号 `pressed`，`connect` 就是把信号和函数绑在一起。点了按钮 → 信号触发 → 函数执行。

### AutoLoad 是什么？

`PlayerActionManager` 是 AutoLoad（全局单例），任何脚本都能直接写名字访问，不需要 `get_node` 层层找。

```gdscript
# ActionCardUI 里直接写
PlayerActionManager.set_selected_action(action)

# LevelScene 里直接写
PlayerActionManager.set_selected_action(move_action)
```

就像公司前台，任何人都能直接打电话，不需要知道前台在哪个办公室。

### 启动时默认选中移动

```gdscript
# LevelScene.gd
func _ready():
    var move = Unit.actions_manager.get_action("move_action")
    PlayerActionManager.set_selected_action(move)  # 默认选移动，不用先点按钮
```

### 状态锁的全局唯一性

```
点了弓箭按钮 → selected_action = 弓箭
点了鼠标 → try_perform_selected_action()
    ├── is_performing_action = true（上锁）
    └── 弓箭.start_action(目标, on_action_finished)

执行中狂点：
    ├── 点其他按钮 → set_selected_action 看到 true → 忽略
    └── 点鼠标 → try_perform 看到 true → 忽略

执行完 → on_action_finished() → is_performing_action = false（解锁）
```

**全局只有一份锁**，不管换了什么技能，正在执行时就锁死。

### 一句话记住第七课

> **Unit 彻底解放，只管"我是谁我在哪"。PlayerActionManager 当全局前台，管"玩家选了什么、点了哪里、要不要执行"。点了按钮 → 前台换技能 → 点了鼠标 → 前台调用当前技能 → 技能干完 → 前台解锁。**

## 🧠 七节课串起来的完整流程

```
【启动时】
地砖厂（NavLayer）醒了
├── 生成导航软件，设置范围/砖大小/不准斜走
├── 扫描所有画了砖的格子
│   └── 每格发一张档案卡（GridData），存进档案柜
│   └── 如果是墙 → 档案卡写 walkable=false，导航标为 solid
└── 向翻译官（GridManager）报户口

角色身上的 ActionsManager 醒了
└── 遍历自己的子节点（MoveAction、BowAction 等），登记到 actions 数组

【运行时】
玩家点 UI 按钮（如"弓箭"）
└── ActionCardUI.on_action_seleted() → PlayerActionManager.set_selected_action(弓箭)
    └── 前台：selected_action = 弓箭

玩家点击鼠标左键
├── PlayerActionManager._unhandled_input
├── try_perform_selected_action()
│   ├── is_performing_action = true（上锁）
│   ├── 翻译官查鼠标在第几块砖
│   └── selected_action.start_action(目标格子, 前台的名片)
│       └── 如果是 MoveAction：
│           ├── super：is_active=true，存好名片
│           └── 算路径 → 存 path
│       └── 如果是 BowAction：
│           ├── super：is_active=true，存好名片
│           └── print("Start 弓箭") → finish_action()
│
├── 技能执行中...（每帧 _process）
│   └── MoveAction：朝 path[0] 移动 → 到了擦脚印
│
└── 技能走空 → finish_action() → 拨通名片
    └── PlayerActionManager.on_action_finished() → is_performing_action=false（解锁）

【UI初始化】
GameUI (CanvasLayer) 醒了
└── UnitActionUI 醒了
    └── call_deferred("update_unit_actions_ui")
        ├── 清空旧按钮（queue_free）
        ├── 遍历 ActionsManager 的所有技能
        │   └── instantiate() 造按钮 → add_child 塞进容器 → set_up 设置名字
        └── 显示：移动 | 弓箭 | 剑击 | 火球

【启动默认】
LevelScene._ready()
└── PlayerActionManager.set_selected_action(移动)  ← 默认选中移动
```

## 📌 记忆锚点（闭眼能想起来的关键词）

| 关键词 | 想起什么 |
|--------|---------|
| `to_local` | 地砖厂可能不在原点，必须转相对坐标 |
| `move_toward` | 朝目标走，到了直接跳到位 |
| `remove_at(0)` | 踩一个脚印擦一个 |
| `AStarGrid2D` | 手机导航软件，算最短路线 |
| `set_point_solid` | 告诉导航：这格是墙，绕开 |
| `GridData` | 每块砖的档案卡 |
| `grid_data_dict` | 档案柜，钥匙是砖号 |
| `_unhandled_input` | UI 先吃，吃剩的才给游戏 |
| `Callable` | 把函数当电话号码存起来，以后 `.call()` 拨打 |
| `self.xxx = xxx` | 参数和变量同名时，加 self 表示存到成员变量 |
| `filter(func)` | 数组筛选，传匿名函数当筛选条件 |
| `get_children()` | 把当前节点下的子节点全翻出来 |
| `BaseAction` | 厨师模板，所有技能继承它 |
| `ActionsManager` | 外卖平台，管登记厨师、按编号找人 |
| `on_action_finished.call()` | 炒完菜拨电话，通知外面"干完了" |
| `super.xxx()` | 调用父类的同名函数，在标准流程上追加 |
| `is_performing_action` | 状态锁，防止玩家狂点导致技能叠发 |
| `unit.global_position` | MoveAction 不是 Unit，通过引用操作角色位置 |
| `is_active` | 保险栓，没激活时 _process 直接 return |
| `extends BaseAction` | 继承厨师模板，获得 start/finish 标准接口 |
| `PackedScene` | 预制件/图纸，存着节点模板 |
| `instantiate()` | 用图纸造出真的实例 |
| `add_child()` | 把节点挂到父节点下面 |
| `queue_free()` | 排队销毁，安全版删除 |
| `call_deferred()` | 延迟到下一帧执行 |
| `CanvasLayer` | UI 专用层，永远渲染在最上面 |
| `HBoxContainer` | 横向自动排列子节点 |
| `PlayerActionManager` | 全局前台，管输入、管技能选择、管状态锁 |
| `set_selected_action` | 切换当前选中的技能 |
| `try_perform_selected_action` | 执行当前选中的技能 |
| `pressed.connect(func)` | 按钮信号连接：点了按钮 → 执行函数 |
| `AutoLoad` | 全局单例，任何脚本直接写名字就能访问 |
| `Unit 瘦身` | 不管输入、不管锁、不管移动，只管"我是谁我在哪" |
