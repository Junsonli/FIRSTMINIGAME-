> 给我自己看的。用大白话 + 生活类比，看完能讲给别人听。

## 🎮 这项目到底在做什么？

做一个**俯视角战棋游戏**。类似《火焰纹章》那种：
- 鼠标悬停在格子上，格子会亮
- 点击空地，角色沿着格子走过去
- 有墙的地方角色绕过去，不会穿墙

## 第一课：鼠标悬停高亮

### 问题是啥？

画面里铺了一堆瓦片（像地砖），鼠标移到哪块砖上，要在那块砖正中央显示一个高亮光圈。

### 难点：鼠标知道的是"像素"，地砖用的是"第几块"

想象你站在一个广场上：
- **像素坐标**："我在东经 116.4° 北纬 39.9°"（很精确，但没法直接说在哪块砖）
- **网格坐标**："我在第 3 行第 5 列的地砖上"（整数，游戏逻辑用的）

所以需要**翻译官**把两种坐标互相转换。

### 三个脚本的分工（像一家餐厅）

| 脚本 | 角色 | 干啥 |
|------|------|------|
| `nav_layer.gd` | **地砖厂** | 铺地砖的，知道每块砖多大 |
| `grid_manager.gd` | **翻译官** | 专门换算"像素 ↔ 第几块砖" |
| `grid_indicator.gd` | **手电筒** | 一个带光圈图片的节点，跟着鼠标跑 |

### 翻译官的换算公式（背下来）

```
像素 → 第几块砖：
  先问地砖厂"鼠标相对于你在哪" → 再除以每块砖的大小
  
  nav_layer.to_local(鼠标像素)  →  nav_layer.local_to_map(结果)

第几块砖 → 像素：
  先乘砖的大小 → 再问地砖厂"相对于整个世界在哪"
  
  nav_layer.map_to_local(砖号)  →  nav_layer.to_global(结果)
```

**为什么要先 `to_local`？**

地砖厂可能不在广场原点（被挪到角落了）。你站在广场上指一个点，想知道是地砖厂的第几块砖，必须先把坐标转成"相对于地砖厂的角度"。

> 现在地砖厂在原点，所以 `to_local` 看起来啥都没做。但以后换关卡、移动地图，没这步就全错了。这叫**防御性编程**。

### 手电筒怎么跟着走？

每帧（60 次/秒）问翻译官：
1. 鼠标现在在第几块砖？
2. 这块砖的像素中心在哪？
3. 把手电筒移到那个像素位置

**优化**：如果鼠标还在同一块砖里晃，砖号没变，手电筒就不动。省性能。

### 一句话记住第一课

> **像素是精确位置，砖号是游戏逻辑。翻译官负责来回换算，手电筒负责显示。**

## 第二课：角色点击移动 + A* 寻路

### 问题是啥？

点击空地，角色要沿着格子走过去。不能直接飞过去，得沿着格子一步一步走，还要走最短路线。

### A* 是什么？

**手机导航软件。**

你输入"从家到公司"，导航软件自动找最短路线。A* 就是游戏里的导航软件，只不过它的"路"是格子。

### 导航软件怎么初始化？（`nav_layer.gd` 里的 `initialize()`）

```
创建导航软件 AStarGrid2D
├── 告诉它地图范围有多大（`get_used_rect()`）
├── 告诉它每块砖多大（16×16 像素）
├── 告诉它能不能斜着走（不能，战棋只走上下左右）
└── `update()` → 生成内部地图数据
```

### 角色移动的三个步骤

**第一步：点击 → 算路线**

```
玩家点击鼠标
  ↓
角色问翻译官：鼠标在第几块砖？
  ↓
角色说：导航软件，从我现在的砖到鼠标那块砖，最短路线是？
  ↓
导航软件返回一串砖号：[砖1, 砖2, 砖3, 砖4]
  ↓
翻译成像素坐标：[像素1, 像素2, 像素3, 像素4]
  ↓
存进 `path` 数组
```

**第二步：每帧沿着路线走**

`path` 就像地上的一排脚印：

```
path = [👣, 👣, 👣, 👣]
         ↑
       当前目标
```

角色每帧盯着第一个脚印 `path[0]` 走，走到后擦掉它（`remove_at(0)`），跟着下一个走。

**第三步：具体移动**

`move_toward(目标, 速度 × 时间差)`

- 朝目标平滑移动
- 如果剩余距离比这一步还小，直接跳到目标（不会冲过头）
- `速度 × 时间差` 保证走多快和电脑卡不卡没关系

### `_unhandled_input` 是干嘛的？

点击分优先级：
1. UI 先吃（点按钮、点菜单）
2. UI 没处理的，才轮到游戏世界

这样点"结束回合"按钮时，角色不会跟着乱走。

### 一句话记住第二课

> **点击 → 导航算路线 → 存成一串像素点 → 每帧踩着脚印走。**

## 第三课：避障 + 格子档案系统

### 问题是啥？

第二课所有格子都能走，角色会穿墙。现在要让墙变成"走不通"。

### 怎么标记墙？

**方法一：给瓦片贴标签**

`NaviData.png` 有两块瓦片：
- 第一块（0:0）贴了标签 `walkable = true` → 地板
- 第二块（1:0）没贴标签 → 默认 `walkable = false` → 墙

在编辑器里用不同瓦片刷地图，地板和墙就自带属性了。

**方法二：给每块砖建档案卡（GridData）**

```
GridData 档案卡模板
├─ walkable: true/false   ← 能走吗？
└─ occupied_unit: 谁站着  ← 有人吗？
```

游戏启动时，遍历所有画了瓦片的格子，给每个格子发一张档案卡，存进 `grid_data_dict`（档案柜）。

```
档案柜 grid_data_dict：
  钥匙：第几块砖（Vector2i）
  内容：那张砖的档案卡（GridData）
```

### 遇到墙怎么办？

启动时检查：如果瓦片的 `walkable = false`：
1. 档案卡写上 `walkable = false`
2. 告诉导航软件：`set_point_solid(这块砖)` → 导航绕开这块砖

```
地图：
┌──┬──┬──┬──┐
│起│  │墙│墙│
├──┼──┼──┼──┤
│  │  │墙│目│
└──┴──┴──┴──┘

导航算出来的路：下 → 下 → 右 → 右 → 上 → 上（绕开墙）
```

### GridManager 新增的查询功能

| 功能 | 干嘛用的 |
|------|---------|
| `is_valid_grid` | 鼠标点地图外面了？返回 false |
| `is_grid_walkable` | 这格能走吗？ |
| `set_grid_walkable` | 动态改（比如门开了），同时更新导航 |
| `is_grid_occupied` | 这格有人吗？（战棋不能两个人站一起） |
| `set_grid_occupied` | 角色移动时，旧格清空，新格登记 |

### 为什么要 GridData + A* 分开？

- **A*（导航软件）**：只管寻路，不知道这格是墙还是水还是陷阱
- **GridData（档案卡）**：存详细信息，以后能扩展：移动力消耗、陷阱伤害、地形加成……

两者解耦：改档案 → 同步更新导航。其他逻辑只查档案，不直接碰导航。

### 一句话记住第三课

> **每块砖一张档案卡，启动时扫描全图建档。墙在档案上标"不可走"，同时让导航绕开。**

## 第四课：技能系统（初步骨架）

### 问题是啥？

第二节课 Unit 直接处理移动，所有逻辑挤在一起。以后加了攻击、施法、喝药……Unit 会变成一团乱麻。

### 解决方案：把"做什么"和"怎么做"分开

| 角色 | 职责 |
|------|------|
| **Unit** | 收到玩家输入，"我要移动" |
| **ActionsManager（外卖平台）** | 登记所有厨师，按编号找人 |
| **BaseAction（厨师模板）** | 统一接口：接单 → 干活 → 炒完通知 |
| **MoveAction/AttackAction（具体厨师）** | 继承 BaseAction，实现各自的干活逻辑 |

### Callable：把函数当电话号码存起来

正常情况下函数是"直接执行"：
```gdscript
func 做菜():
    print("好了")
做菜()  # 立刻执行
```

Callable 是"先存起来，以后拨打"：
```gdscript
var 电话: Callable = 做菜    # 存进通讯录（不执行）
# 一小时后...
电话.call()                   # 现在拨打，执行！
```

**为什么要存起来？**

厨师（BaseAction）炒菜时，外卖平台（ActionsManager）不想一直盯着。平台给了厨师一个电话，厨师炒完自己拨过去。

### BaseAction：厨师模板

```gdscript
class_name BaseAction

@export var action_id:String        # 厨师编号，如 "move"
@export var action_name:String      # 厨师名字，如 "移动"
var unit:Unit                       # 在哪个店（Unit）工作
var is_active:bool = false          # 正在炒菜吗？
var on_action_finished:Callable     # 炒完打这个电话
```

**start_action（接单）：**
```gdscript
func start_action(目标格子, 电话号码:Callable):
    is_active = true
    self.on_action_finished = 电话号码   # 存进通讯录
```

**finish_action（通知）：**
```gdscript
func finish_action():
    is_active = false
    on_action_finished.call()          # 拨打存好的电话
```

### ActionsManager：外卖平台

**开店时登记厨师：**
```gdscript
func _ready():
    for 厨师 in get_children():        # 遍历所有子节点
        actions.append(厨师)           # 登记到花名册
```

**按编号找厨师：**
```gdscript
func get_action(action_id):
    var results = actions.filter(func(a): return a.action_id == action_id)
    if not results.is_empty():
        return results[0]              # 返回找到的厨师
    return null
```

### 为什么加 self.？

```gdscript
self.on_action_finished = on_action_finished
```

参数名和成员变量名一样，都是 `on_action_finished`。不加 `self` 等于"把参数赋值给参数"，什么都没发生。加了 `self` 才是"把参数存到成员变量"。

### 一句话记住第四课

> **Unit 只管点菜，ActionsManager 管找厨师，厨师炒完自己打电话。新增技能 = 新增一个厨师，不改平台。**

## 第五课：移动技能 + 其他技能模板

### 问题是啥？

第四课搭了空骨架，Unit 还是自己在走路。现在要把移动逻辑真正拆进 MoveAction，让 Unit 只负责"喊开工"。

### Unit 以前 vs 现在

**以前（第二课）：**
```
Unit 自己管一切：
├── 收到点击
├── 算路径
├── 每帧走路
└── 擦脚印
```

**现在（第五课）：**
```
Unit 只管喊开工：
├── 收到点击
├── is_performing_action = true（上锁，防止连点）
├── actions_manager.get_action("move_action").start_action(目标, 回调)
└── 干等

MoveAction 管走路：
├── start_action 里算路径
├── _process 每帧走路
└── 走完了 finish_action() → 拨通 Unit 的电话

Unit 接电话：
└── on_action_finished() → is_performing_action = false（解锁）
```

### 状态锁 `is_performing_action`

战棋是**一个指令执行完才能下另一个**。

```
玩家狂点鼠标：
第1次点击 → 开始移动 → is_performing_action = true
第2次点击 → _unhandled_input 看到 true → 直接 return（屏蔽）
第3次点击 → 同上，被屏蔽
...
移动结束 → on_action_finished() → is_performing_action = false → 才能接受下一次
```

### `super.start_action(...)` 是干嘛的？

父类 BaseAction 已经写了开工的标准流程：
```gdscript
func start_action(target, on_action_finished):
    is_active = true
    self.on_action_finished = on_action_finished
```

MoveAction 不想重写一遍，所以：
```gdscript
func start_action(target, on_action_finished):
    super.start_action(target, on_action_finished)   # 先执行父类的标准流程
    path = GridManager.get_nav_world_path(...)        # 再追加自己的逻辑（算路径）
```

**`super` = "调用父类的同名函数"。**

### `unit.global_position`

MoveAction 不是 Unit 本身，是挂在 ActionsManager 下的独立节点。所以要通过 `unit` 引用（BaseAction 从 `owner` 获取的）来操作角色的位置。

```
MoveAction 节点 ──unit──→ Unit 节点
```

### `_process` 里的 `if not is_active: return`

保险栓。只有 `start_action` 被调用后（`is_active = true`），MoveAction 才会执行移动逻辑。平时进来直接 return，什么都不做。

### 其他技能模板

BowAction、SwordAction、FireballAction 目前只打印一行字就结束。但它们的结构已经搭好了：

```gdscript
extends BaseAction
class_name BowAction

func start_action(target, on_action_finished):
    super.start_action(target, on_action_finished)
    # 以后在这里写：播放动画、生成箭矢、检测命中、扣血
    finish_action()
```

**以后把 `print` 换成实际逻辑就行，Unit 和 ActionsManager 完全不用改。**

### Callable 名片的本质

Unit 传给 MoveAction 的不是"函数的名字"，而是一张**包含回家地址的名片**：

```
┌─────────────────────────┐
│  Callable 名片           │
├─────────────────────────┤
│ 对象: Unit (self)       │  ← 回家地址
│ 函数: on_action_finished │  ← 电话号码
└─────────────────────────┘
```

MoveAction 存好名片，走完后 `on_action_finished.call()` → Godot 按名片上的地址找到 Unit，在其身上执行函数。

**不是跨脚本传播，是函数引用自带对象地址。**

### 一句话记住第五课

> **Unit 上锁→喊开工→MoveAction 走路→走完了拨名片上的电话→Unit 解锁。新增技能改自己的脚本就行，外面不动。**

## 🧠 五节课串起来的完整流程

```
【启动时】
地砖厂（NavLayer）醒了
├── 生成导航软件，设置范围/砖大小/不准斜走
├── 扫描所有画了砖的格子
│   └── 每格发一张档案卡（GridData），存进档案柜
│   └── 如果是墙 → 档案卡写 walkable=false，导航标为 solid
└── 向翻译官（GridManager）报户口

角色身上的 ActionsManager 醒了
└── 遍历自己的子节点（MoveAction、BowAction 等），登记到 actions 数组

【运行时】
玩家点击空地
├── Unit 检查：is_performing_action? → true? 忽略（防止连点）
├── is_performing_action = true（上锁）
├── 翻译官查鼠标在第几块砖
├── ActionsManager：找 action_id="move_action" 的厨师
├── 厨师.start_action(目标格子, Unit的名片)
│   ├── super：is_active=true，存好名片
│   └── 算路径：nav → 格子路径 → 像素路径 → 存 path
├── 厨师每帧 _process：is_active? → 朝 path[0] 移动 → 到了擦脚印
└── 厨师 path 走空 → finish_action() → 拨通名片 → Unit.on_action_finished() → is_performing_action=false（解锁）
```

## 📌 记忆锚点（闭眼能想起来的关键词）

| 关键词 | 想起什么 |
|--------|---------|
| `to_local` | 地砖厂可能不在原点，必须转相对坐标 |
| `move_toward` | 朝目标走，到了直接跳到位 |
| `remove_at(0)` | 踩一个脚印擦一个 |
| `AStarGrid2D` | 手机导航软件，算最短路线 |
| `set_point_solid` | 告诉导航：这格是墙，绕开 |
| `GridData` | 每块砖的档案卡 |
| `grid_data_dict` | 档案柜，钥匙是砖号 |
| `_unhandled_input` | UI 先吃，吃剩的才给游戏 |
| `Callable` | 把函数当电话号码存起来，以后 `.call()` 拨打 |
| `self.xxx = xxx` | 参数和变量同名时，加 self 表示存到成员变量 |
| `filter(func)` | 数组筛选，传匿名函数当筛选条件 |
| `get_children()` | 把当前节点下的子节点全翻出来 |
| `BaseAction` | 厨师模板，所有技能继承它 |
| `ActionsManager` | 外卖平台，管登记厨师、按编号找人 |
| `on_action_finished.call()` | 炒完菜拨电话，通知外面"干完了" |
| `super.xxx()` | 调用父类的同名函数，在标准流程上追加 |
| `is_performing_action` | 状态锁，防止玩家狂点导致技能叠发 |
| `unit.global_position` | MoveAction 不是 Unit，通过引用操作角色位置 |
| `is_active` | 保险栓，没激活时 _process 直接 return |
| `extends BaseAction` | 继承厨师模板，获得 start/finish 标准接口 |
